## 现有架构综述

Perelegans 已经构建了一个很扎实的三层传感+记忆+AI 管线：

| 层 | 能力 | 当前状态 |
|---|---|---|
| **传感层** | Win32 前台窗口采样、进程名/路径/停留时长追踪、截图捕捉 | 基础扎实，5 秒间隔轮询 |
| **记忆层** | 鱼骨/RAG 记忆节点，含类型/标签/权重/plan 状态/星座分类/生命周期/AI 描述与解释/下一步预测 | 结构丰富，已具备 plan 对比和能力衰减 |
| **AI 层** | 专注判断、记忆提取、任务解析、日报生成、桌面洞察（回放/进度推断/鱼骨归因/星图解释/回到现场）、个性化回复、截图分析、上下文压缩 | 功能覆盖面广，prompt 设计成熟 |
| **交互层** | 悬浮气泡提示、主面板对话+星图+鱼骨、专注模式状态管理 | 可用，但偏被动 |

**当前的核心局限**：整个系统偏"被动记录"，缺少三个维度的能力——**时序模式挖掘**、**主动干预**、**跨会话的持续推理**。

## 真正解决痛点的创新方向

基于本地上下文（进程流）+ 记忆（知识图谱）+ 背景感知（你在干什么）这个三件套，我梳理了以下方向，按"痛点严重程度 × 技术可行性 × 现有基础可复用程度"排序：

---

### 1. 智能分心预警 + 意图衰减追踪

**痛点**：专注模式开启后，用户不知不觉漂移到浏览器/社交软件，等回过神来已经浪费 30 分钟。现有的悬浮气泡提示太软弱——"我先轻轻提醒一下"这种语气，用户看多了就免疫。

**方案**：在 `FloatingPetViewModel.ApplyFocusModeBubble` 的基础上，引入三级升级机制：

- **L1（温和）**：3 分钟分心 → 气泡提示 "你在浏览器停了 3 分钟，要回到「写论文」吗？"
- **L2（具体）**：8 分钟分心 → 气泡显示 AI 生成的针对当前 plan 下一步的具体指令，例如 "论文还剩摘要没写完，打开 Word 先列三段结构"
- **L3（阻断）**：15 分钟分心 → 弹出小型 toast，包含一个"我回来了"按钮，点击后自动激活目标应用窗口

同时引入**意图衰减追踪**：如果一个 plan 超过 N 天没有被任何相关进程事件触碰（通过 `QueueSceneCheckpoint` 的 `CalculateProcessPlanAffinity` 检测），主动提醒用户"「学 Rust」这个 plan 已经 5 天没动静了，要不要放弃或重新评估？"

**现有基础**：`FocusModeService`、`ProcessMonitorService.SampleForegroundWindowFocus`、`ApplyFocusModeBubble`、`IsProcessRelevantToFocus` 均已就绪。只需加一个计时器状态机 + AI 生成 L2 提示。

---

### 2. 进程流 → 工作流自动识别

**痛点**：用户每天重复相同的应用切换模式（IDE→Terminal→浏览器→IDE），但这些模式从未被显式捕获为知识。当用户被打断后想"继续刚才"，只知道最后一个应用，不知道整个工作流上下文。

**方案**：对 `ApplicationUsageSession` 的时间序列做模式挖掘：

- 用滑动窗口检测 N-gram 进程序列：`[Code, Terminal, msedge:localhost]` → 标记为"本地开发工作流"
- 当相同序列重复出现 ≥ 3 次，自动创建一条工作流类型记忆："开发 Perelegans 时的标准流程：打开 VS Code → 终端 → 本地调试页"
- 当用户说"继续工作"时，不仅恢复最后场景，而是恢复整个工作流上下文：最近打开的文件、最近的 plan、典型下一步

**现有基础**：`ApplicationUsageSession` 已经按 `StartTime`/`EndTime` 记录每次进程切换的精确时间线，`DatabaseService` 有 `GetApplicationUsageSessionsSinceAsync`。缺的是滑动窗口分析 + 工作流记忆自动创建逻辑。

---

### 3. 精力节律分析 + 主动建议

**痛点**：用户不知道自己什么时候专注力最好，什么时候最容易分心。现有的日报只汇总"今天做了什么"，但缺乏跨天的趋势分析——这恰恰是最有价值的洞察。

**方案**：基于累计的 ApplicationUsage 和 FocusAssessmentResult 数据构建用户画像：

- **黄金时段检测**：分析过去 7/30 天每小时段的生产力应用占比，找到高峰（如 "9:00-11:00 生产力占比 87%"）和低谷（"15:00-16:00 仅 42%"）
- **最优会话时长**：分析连续专注 session 的平均持续时间，建议番茄钟时长 "你平均 38 分钟后会分心，建议试 35 分钟番茄钟"
- **分心源排行**：统计非生产力应用的出现频率和时长
- **周趋势对比**："本周相比上周，代码编辑时间增加 23%，社交媒体时间减少 40%"

在悬浮气泡和日报中主动推送这些洞察："今天是周二，你周二通常在 10 点半后效率下降，要不要趁现在把最难的 plan 先启动？"

**现有基础**：数据库中有完整的时间序列数据，`ApplicationUsage` 有 `TotalDuration` 和 `Category`，`ApplicationUsageSession` 有精确的 `StartTime`/`EndTime`。缺少的是聚合分析 + AI 生成洞察的编排。

---

### 4. 记忆自动关联 + 知识图谱演化

**痛点**：当前记忆节点孤立存储，即使它们在语义上高度相关（"决定使用 SQLite"→"数据库服务实现"→"备份恢复功能"），彼此之间没有显式链接。星图展示的只是空间排布，不是真实关系。

**方案**：周期性（如每次日报时）运行记忆关联分析：

- AI 扫描最近更新的 30-50 条记忆，识别因果关系：决策 → 影响的项目 → 派生的任务 → 产生的工作流
- 创建 `FocusTaskLink` 或类似的记忆关联边：`[记忆 A] -[决定使用]-> [记忆 B] -[实现了]-> [记忆 C]`
- 在星图中，相关的记忆节点用连线表示，hover 显示关系类型
- 查看一条记忆时，系统自动展示 "相关记忆" 链

**现有基础**：`FocusTaskLink` 模型已经定义（任务关联关系），`ContextMemory` 有丰富的元数据字段。缺的是 AI 驱动的批量关联分析 + 可视化连线。

---

### 5. 主动场景存档 + "时间旅行"

**痛点**：现有的 `QueueSceneCheckpoint` 每 30 分钟且应用不同时才触发一次，记录很稀疏。当用户想回顾"昨天下午在干什么"时，只能看到最后一个进程名。缺少时间维度的深度回放。

**方案**：

- **高频存档**：每 12 分钟 + 应用切换时自动保存一个轻量场景快照（进程名、时长、关联 plan、当前时间），不调用 AI，纯本地存储
- **AI 摘要分层**：每天结束时，AI 把当天所有快照压缩成一段叙事时间线："下午 2:15-3:00 在 VS Code 修改 App.xaml.cs，3:00-3:12 短暂切换到浏览器查文档，3:12-4:30 回到 VS Code 继续"
- **时间旅行 UI**：主面板增加一个时间轴滑块，拖动可看到每个时间段的摘要，点击展开详情
- **跨天对比**："和昨天下午对比，你今天多花了 23 分钟在浏览器上"

**现有基础**：`QueueSceneCheckpoint` 框架已存在，`ForegroundFocusSnapshot` 有完整的进程信息。需要的是更高频的快照存储 + AI 时间轴压缩 + 时间旅行 UI。

---

### 6. 上下文触发的记忆注入

**痛点**：用户和 Perelegans 对话时，经常需要手动描述上下文——"我刚才在写论文"。系统其实已经知道用户在什么应用、停留了多久、关联什么 plan，但个性化回复中这些信息的使用偏弱。

**方案**：

- 增强 `ContextRetrievalService.RetrieveAsync`：不仅做 BM25 关键词匹配，还根据当前进程名和活跃 plan 标签做语义扩展。例如当前在 Word 里且 plan 标签含 "writing,paper"，自动提升"论文"相关记忆的权重
- 在 `CreatePersonalizedReplyAsync` 的 prompt 中，显式注入 "当前用户正在 X 应用中停留了 Y 分钟，最近的 plan 是 Z，上一次场景检查点发现……"
- 当用户问"这段代码写得怎么样"，系统能推断出用户在 VS Code 且最近打开的是 C# 项目，回复自然带有项目上下文

**现有基础**：`ForegroundFocusSnapshot` 已传入 `CreatePersonalizedReplyAsync`，但 prompt 中使用不够充分。调整 prompt + 增强检索即可。

---

### 7. 本地优先的定时任务引擎

**痛点**：日报需要手动触发（点按钮或输入"复盘"），不能自动在每天 18:00 生成。用户可能忘记做日报，导致数据只积累不消化。

**方案**：

- 增加一个简单的本地定时器服务，支持配置触发时间（如每天 18:00、每周日 20:00）
- 自动触发日报/周报生成，保存到本地 Markdown 文件并可选推送到托盘通知
- 支持条件触发："当今天的生产力应用时长超过 4 小时时，自动生成日报"
- 支持预设计划提醒："明天 9:00 提醒我继续「写论文」"

**现有基础**：`GenerateDailyReview` 逻辑完整，`DispatcherTimer` 框架可用。需要的是一个持久化的定时任务配置 + 调度器。

## 实施优先级建议

如果只选 3 个先做，我的建议是：

1. **精力节律分析（#3）**—— 最能给用户"这东西真有用"的体感，数据已经全了，只需要分析层
2. **智能分心预警升级（#1）**—— 直接提升专注模式的有效性，改动集中在 FloatingPetViewModel 和 FocusModeService
3. **工作流自动识别（#2）**—— 发挥进程监控数据中隐藏的最大价值，把"被动记录"变为"主动理解"

这三个都不需要改动数据模型 Schema，主要在服务层和 ViewModel 层增加逻辑，风险低、见效快。
